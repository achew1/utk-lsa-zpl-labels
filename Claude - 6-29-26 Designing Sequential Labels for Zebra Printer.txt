## How Sequential Zebra Labels Work

There are **two main approaches** depending on your use case:

### Approach 1: ZPL `^SN` (Serial Number) — Built-in Auto-Increment
The `^SN` command tells the printer to auto-increment a field across multiple label copies[doc1]. You pair it with `^PQ` (print quantity) and the printer handles the counting internally[doc2].

**Syntax:** `^SNs,i,p`
- `s` = starting value (e.g., `001`)
- `i` = increment step (e.g., `1`)
- `p` = pad with leading zeros (`Y` or `N`)

**Best for:** A single field that increments linearly (e.g., Shelf 001, 002, 003...)[doc2][doc1]

### Approach 2: Generate Labels Programmatically
For **multi-field sequential logic** (like Range + Ladder + Shelf all incrementing with nested rollover), you generate one ZPL block per label and concatenate them into a single print job[doc3]. This gives you full control over the logic.

**Best for:** Complex sequences like Range A → Ladder 1–10 → Shelf 1–5 (nested counters)[doc3]

---

## Recommended Strategy for Range / Ladder / Shelf

Since your labels have **three interdependent sequential fields**, the programmatic approach is the most reliable. Here's the logic:

- **Shelf** increments on every label (1 → 2 → 3...)
- **Ladder** increments when Shelf rolls over
- **Range** increments when Ladder rolls over

## How Sequential Zebra Labels Work

Zebra printers use **ZPL (Zebra Programming Language)**[doc4] to define label content and layout. For sequential labels, you have two main approaches:

### Option 1 — Built-in `^SN` Command (Simple Single Counter)
ZPL has a native `^SN` (Serial Number) command[doc1] that auto-increments a value across copies when combined with `^PQ` (print quantity):

```zpl
^XA
^FO30,20^A0N,40,40^FDShelf: ^FS
^FO200,20^SN001,1,Y^FS   ← starts at 001, increments by 1, zero-padded
^PQ50                     ← print 50 copies
^XZ
```
This is great for **one sequential field**, but gets complex when you need three fields (Range, Ladder, Shelf) that roll over into each other.

---

### Option 2 — Pre-generated ZPL (Recommended for Multi-Field)
For **Range + Ladder + Shelf** with nested rollover (e.g., Shelf 1–5 per Ladder, Ladders 1–10 per Range), the cleanest approach is to **generate all labels programmatically** and concatenate them into one ZPL print job[doc3]. Each label is its own `^XA...^XZ` block.

The Python script above does exactly this. Here's what each label looks like:

```
┌──────────────────────────────────┐
│  Range: A       Ladder: 01       │
│         Shelf: 001               │
│  ||||||||||||||||||||||||||||     │
│       A-01-001  (barcode)        │
└──────────────────────────────────┘
```

## Key ZPL Commands Used[doc4]

| Command | Purpose | Example |
|---|---|---|
| `^XA` / `^XZ` | Start / end label | — |
| `^FO x,y` | Set field position (x, y in dots) | `^FO30,20` |
| `^FD` / `^FS` | Field data / field separator | `^FDRange: A^FS` |
| `^CF0,40` | Set font size (40 dots tall) | — |
| `^BCN,80,Y,N,N` | Code 128 barcode | — |
| `^PW` / `^LL` | Label width / length in dots | `^PW812` |
| `^SN` | Auto-increment serial number | `^SN001,1,Y` |

Great question! When you run `python zebra_sequential_labels.py` without any flags, the script does the following:

---

## What Happens Under the Hood

### 1 — The Loop
The script runs through every combination of Ladder and Shelf for the configured range side:
```python
for ladder in range(1, 59):        # 1 through 58
    for shelf in range(1, 16):     # 1 through 15
        all_zpl += build_label("5L", ladder, shelf)
```

### 2 — Each Label is Built as a ZPL Text Block
The `build_label()` function returns a ZPL string (just text!) for each label — `^XA ... ^XZ`. All 870 of those strings are concatenated together into one big block of text.

### 3 — That Text is Written to a File
```python
with open("labels_5L.zpl", "w") as f:
    f.write(all_zpl)
```
That's it — a `.zpl` file is just a **plain text file** with ZPL commands inside. There's nothing special about the format — Zebra printers simply know how to read that text and turn it into printed labels.

---

## So Why Not Just Print Directly?
If `PRINT_DIRECTLY = False`, the script saves the file locally and **you** are responsible for sending it to the printer. If `PRINT_DIRECTLY = True`, the script takes that same text and streams it directly to the printer over the network via its IP address instead of saving it — same content, different destination.

Does that clear it up? 😊

That's a very interesting bug! The ZPL itself is correct — all shelf numbers use the same Y coordinate (`79`), so the script isn't doing anything differently for those numbers. This points to a **ZPL font rendering quirk**.

---

## What's Likely Happening

The `^A0N` font in ZPL is a scalable font. The digit `1` is significantly **narrower** than other digits — and certain combinations like `11`, `14`, and `15` can cause the font renderer to position the glyph block slightly differently due to how character metrics are calculated internally.

---

## Before Fixing — Two Quick Questions

**1 — Is this in the online preview only, or on actual printed labels too?**
The online rendering tools (both Labelary and zplprinter) are third-party renderers and don't always match exactly how a real Zebra printer renders. The actual printer might not have this issue at all.

**2 — Does it look "higher" or just visually different?**
Because `1` is narrower, `11`, `14`, `15` may just *look* higher due to the whitespace on either side — an optical illusion rather than an actual position difference.

---

## If It's a Real Issue on the Printer

The fix would be to switch from `^A0N` to `^CF0` (font 0, also scalable) which handles character metrics differently:

```zpl

^PW402
^LL203
^LH0,0

^FO271,0^GB1,203,1^FS
```
